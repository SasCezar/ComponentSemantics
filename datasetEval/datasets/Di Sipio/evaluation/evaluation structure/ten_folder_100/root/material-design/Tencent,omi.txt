    }\n    }\n\n    define(\'my-counter\', _ => html`\n      <div>\n        <button onClick=${_.store.sub}>-</button>\n        <span>${_.store.data.count}</span>\n        <button \n      css: `span { color: red; }`,\n      installed() {\n        console.log(\'installed\')\n      }\n    })\n\n    render(html`<my-counter />`, \'body\', new Store)\n  </script>\n</body>\n\n</html>\n```\n\nOmi Store provides a way to pass data through the component tree without having to pass props down manually at every level, injected from the root component and Install\n\n```bash\n$ npm i omi-cli -g    # install cli\n$ omi init my-app     # init project\n$ cd my-app           \n$ npm start           # develop\n$ npm run build       # release\n```\n\n> `npx omi-cli init my-app` is also supported(npm v5.2.0+).\n\nDirectory description:\n\n```\n\xe2\x94\x9c\xe2\x94\x80 config\n\xe2\x94\x9c\xe2\x94\x80 public\n\xe2\x94\x9c\xe2\x94\x80 \xe2\x94\x9c\xe2\x94\x80 store       //Store all this store of pages\n\xe2\x94\x82  \xe2\x94\x9c\xe2\x94\x80 admin.js    //Entry js of compiler\xef\xbc\x8cwill build to admin.html\n\xe2\x94\x82  \xe2\x94\x94\xe2\x94\x80 index.js    //Entry js of compiler\xef\xbc\x8cwill build to index.html\n```\n\n#### Scripts\n\n```json\n"scripts": {\n    "start": "node scripts/start.js",\n    "build": extends WeElement {\n  onClick = evt => {\n    // trigger CustomEvent\n    this.fire(\'abc\', { name: \'dntzhang\', age: 12 })\n    evt.stopPropagation()\n  }\n\n  //If you need to use <hello-element></hello-element> directly in html, you must declare propTypes\n  static propTypes = {\n    msg: String\n  }\n\n  static css = `\n      div {\n        color: red;\n        cursor: pointer;\n      }`\n\n  render(props) {\n    return (\n      <div onClick={this.onClick}>\n        Hello {props.msg}\n        <div>Click Me!</div>\n      </div>\n    )\n  }\n})\n```\n\nUsing `hello-element`:\n\n```js\nimport { define, render, WeElement } from \'omi\'\nimport \'./hello-element\'\n\ndefine(\'my-app\', class extends WeElement {\n  data = { abc: \'abc\' }\n\n  // define CustomEvent Handler\n  onAbc = evt => {\n    // get evt data by evt.detail\n    this.data.abc = \' by \' + evt.detail.name\n    this.update()\n  }\n\n  static css = `\n      div{\n          color: green;\n      }`\n  }\n\n  render(props, data) {\n    return (\n      <div>\n        Hello {data.abc}\n        <hello-element\n          onAbc={this.onAbc}\n          msg="WeElement"\n        />\n      </div>\n    )\n  }\n})\n\nrender(<my-app name="Omi v4.0" />, \'body\')\n```\n\nTell Babel to transform JSX into `Omi.h()` call:\n\n```json\n{\n  "presets": ["env", "omi"]\n}\n```\n\nThe following two NPM packages need to be installed to support the above configuration:\n\n```bash\n"babel-preset-env": "^1.6.0",\n"babel-preset-omi": "^0.1.1",\n```\n\nIf you use babel7, you can also use the following packages and configuration:\n\n```bash\nnpm install --save-dev @babel/preset-env\nnpm install --save-dev @babel/preset-react\n```\n\n```js\n{\n  "presets": [\n    "@babel/preset-env",\n    [\n      "@babel/preset-react",\n      {\n        "pragma": "Omi.h",\n        "pragmaFrag": "Omi.h.f"\n      }\n    ]\n  ]\n}\n```\n\nIf you don\'t want to write CSS in JS, you can use [to-string-loader](https://www.npmjs.com/package/to-string-loader) of webpack,\nFor example, the following configuration:\n\n```js\n{\n  test: /[\\\\|\\/]_[\\S]*\\.css$/,\n  use: [\n    \'to-string-loader\',\n    \'css-loader\'\n  ]\n}\n```\n\nIf your CSS file starts with "`_`", CSS will use `to-string-loader`, such as:\n\n``` js\nimport { tag, WeElement render } from \'omi\'\n\ndefine(\'my-app\', class extends WeElement {\n\n  css = require(\'./_index.css\')\n  ...\n  ...\n  ...\n```\n\nYou can also forget the tedious configuration and use omi-cli directly, no together.\n\n## Design philosophy\n\nThe Omi design was driven by The Zen of Python, by Tim Peters philosophy:\n\n* Beautiful is better than ugly.\n* Explicit is better than implicit.\n* Simple is better than complex.\n* Complex is better than complicated.\n* Flat is better than nested.\n* Sparse is better than dense.\n* Readability counts.\n* Special cases aren\'t special enough to break the rules.\n* Although practicality beats purity.\n* Errors should never pass silently.\n* Unless explicitly silenced.\n* In the face of ambiguity, refuse the temptation to guess.\n* There should be one\xe2\x80\x94and preferably only one\xe2\x80\x94obvious way to do it.\n* Although that way may not be obvious at first unless you\'re Dutch.\n* Now is better than never.\n* Although never is often better than right now.\n* If the implementation is hard to explain, it\'s a bad idea.\n* If the implementation is easy to explain, it may be a good idea.\n* Namespaces are one honking great idea\xe2\x80\x94let\'s do more of those!\n\n\n## Core Maintainers\n\n- [@Wscats](https://github.com/Wscats)\n- [@f](https://github.com/f)\n- [@LeeHyungGeun](https://github.com/LeeHyungGeun)\n-
