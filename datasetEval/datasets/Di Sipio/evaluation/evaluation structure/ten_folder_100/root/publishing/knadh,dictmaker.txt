b'# dictmaker\n![dictmaker](https://user-images.githubusercontent.com/547147/63648814-ade19a80-c752-11e9-9c63-cbf5dfb2269e.png)\n\n**dictmaker** is a stand-alone, single-binary server application for building and publishing dictionary websites. [Alar](https://alar.ink) (Kannada-English dictionary) is an example in production.\n\n- Generic entry-relation (many)-entry structure for dictionary data in a Postgres database with just two tables (entries, relations)\n- Entirely built on top of Postgres full text search using [tsvector](https://www.postgresql.org/docs/10/datatype-textsearch.html) tokens\n- Works with any language. Plug in external tokenizers or use a built in tokenizer supported by Postgres for search\n- Possible to have entries and definitions in any number of languages in a single database\n- HTTP/JSON REST like APIs\n- Themes and templates to publish dictionary websites\n- Paginated A-Z (or any alphabet for any language) glossary generation for dictionary words\n\n## How it works\ndictmaker has no concept of language or semantics. To make a universal dictionary interface possible, it treats all data as unicode strings that can be searched with Postgres DB\'s fulltext capabilities, where search tokens are generated and stored along with the dictionary entries. There are several built-in fulltext dictionaries and tokenizers that Postgres supports out of the box, mostly European languages (`\\dFd` lists installed dictionaries). For languages that do not have fulltext dictionaries, it is possible to generate search tokens manually using an external algorithm. For instance, to make a phonetic English dictionary, [Metaphone](https://en.wikipedia.org/wiki/Metaphone) tokens can be generated manually and search queries issued against them.\n\n### Concepts\n- There can be any number of languages defined in the dictionary. eg: \'english\', \'malayalam\', \'kannada\' etc.\n- All content, the entry words and their definitions, are stored in the `entries` table\n- Entry-definition many-to-many relationships are stored in the `relations` table, represented by `from_id` (entry word) -> `to_id` (definition word), where both IDs refer to the `entries` table.\n\n## entries table schema\n| `id`      | `SERIAL`   |                                                                                                                                     |\n|-----------|------------|-------------------------------------------------------------------------------------------------------------------------------------|\n| `guid`    | `TEXT`     | A custom, unique GUID for every entry, like an MD5 hash                                                                             |\n| `content` | `TEXT`     | Actual language content. Dictionary word or definition entries                                                                      |\n| `initial` | `TEXT`     | The first "alphabet" of the content. For English, for the word `Apple`, the initial is `A`                                          |\n| `weight`  | `INT`      | An optional numeric value to order search results                                                                                   |\n| `tokens`  | `TSVECTOR` | Fulltext search tokens. For English, Postgres\' built-in tokenizer gives `to_tsvector(\'fully conditioned\')` = `\'condit\':2 \'fulli\':1` |\n| `lang`    | `TEXT`     | String representing the language of content. Eg: `en`, `english`                                                                    |\n| `types`   | `TEXT[]`   | Strings describing the types of content. Eg `{noun, propernoun}`                                                                    |\n| `tags`    | `TEXT[]`   | Optional tags                                                                                                                       |\n| `phones`  | `TEXT[]`   | Phonetic (pronunciation) descriptions of the content. Eg: `{ap(\xc9\x99)l, aapl}` for `Apple`                                              |\n| `notes`   | `TEXT`     | Optional text notes                                                                                                                 |\n## relations table schema\n| `from_id` | `INT`    | ID of the head word or the dictionary entry in the entries table       |\n|-----------|----------|------------------------------------------------------------------------|\n| `to_id`   | `INT`    | ID of the definition content in the entry table                        |\n| `types`   | `TEXT[]` | Strings describing the types of this relation. Eg `{noun, propernoun}` |\n| `weight`  | `INT`    | An optional numeric value to order definition results                  |\n| `tags`    | `TEXT[]` | Optional tags                                                          |\n| `notes`   | `TEXT`   | Optional text notes                                                    |\n\n# Installation\n1. Download the latest release [release](https://github.com/knadh/dictmaker/releases) of dictmaker\n2. Run `./dictmaker --new` to generate a sample config.toml and DB schema.sql\n3. Create a Postgres DB and execute `schema.sql` on it to create the tables\n4. Define your dictionary\'s languages and properties along with other configuration in `config.toml`\n5. Populate the `entries` and `relations` tables with your dictionary data. See the "Sample dictionary" section below\n6. Run the binary: `./dictmaker`\n \n## Dictionary query API\n```shell\n# /dictionary/from_lang/to_lang/word\n# Optional query params: ?type=noun&type=noun2&tag=a&tag=b ...\ncurl localhost:8080/dictionary/english/english/apple\n```\n\n## Sample dictionary\nThe `sample/sample.sql` shows how to setup an English-English and English-Italian dictionary. Retaining the English-Italian config in the generated sample config file, execute `sample/sample.sql` on your database.\n\nThen try:\n```shell\ncurl localhost:8080/dictionary/english/english/apple\ncurl localhost:8080/dictionary/english/italian/apple\n```\n\n## Themes\nSee the [dictmaker-alar](https://github.com/knadh/dictmaker-alar) repository that powers the [Alar](https://alar.ink) dictionary. A theme is a directory with a collection of Go HTML templates. Run a theme by passing `./dictmaker --site=theme_dir`.\n\n## Tokenizer plugins\nFor languages that do not have Postgres fulltext dictionaries and tokenizers, dictmaker supports loading compiled Go tokenizer plugins that implement the `search.Tokenizer` interface. See `tokenizers/kannada` (and `Makefile` for compilation help).\n\nLicensed under the AGPL v3 license.\n'
