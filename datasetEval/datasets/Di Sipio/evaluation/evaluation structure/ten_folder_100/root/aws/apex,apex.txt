b'\n# No longer maintained\n\nThis software is no longer being maintainted and should not be chosen for new projects, unless your company is interested in sponsoring the project it\'s unfortunately unsustainable. See this [issue](https://github.com/apex/apex/issues/932) for more information\n\n![Apex Serverless Architecture](assets/title.png)\n\nApex lets you build, deploy, and manage [AWS Lambda](https://aws.amazon.com/lambda/) functions with ease. With Apex you can use languages that are not natively supported by AWS Lambda through the use of a Node.js shim injected into the build. A variety of workflow related tooling is provided for testing functions, rolling back deploys, viewing metrics, tailing logs, hooking into the build system and more.\n\nThis project is designed for event-driven pipelines as it does not abstract away FaaS (functions as a service). If you are building web applications, APIs, or sites, consider using [Apex Up](https://github.com/apex/up), which provides a more out-of-the-box experience for these use-cases.\n\n## Installation\n\nOn macOS, Linux, or OpenBSD run the following:\n\n```\ncurl https://raw.githubusercontent.com/apex/apex/master/install.sh | sh\n```\n\nNote that you may need to run the `sudo` version below, or alternatively chown `/usr/local`:\n```\ncurl https://raw.githubusercontent.com/apex/apex/master/install.sh | sudo sh\n```\n\nOn Windows download [binary](https://github.com/apex/apex/releases).\n\nAfter downloading, rename binary file \'apex.exe\', then add to PATH.\n\nIf already installed, upgrade with:\n\n```\napex upgrade\n```\n\n## Runtimes\n\nCurrently supports:\n\n- Node.js\n- Golang\n- Python\n- Ruby\n- Java\n- Rust\n- Clojure\n\nExample projects for all supported runtimes can be found in [_examples](_examples) directory.\n\n## Features\n\n- Supports languages Lambda does not natively support via shim\n- Binary install (install apex quickly for continuous deployment in CI etc)\n- Hook support for running commands (transpile code, lint, dependency management, etc)\n- Batteries included but optional (opt-in to higher level abstractions)\n- Environment variable population via command-line, file, or inline config\n- Idempotent deployments (checksums skip already-deployed code)\n- Multiple environments via `project.ENV.json` and `function.ENV.json` files\n- Configuration inheritance and overrides\n- Command-line function invocation with JSON streams\n- Command & function name autocompletion\n- Function name globbing (ex: `apex deploy api_*`)\n- Transparently generates a zip for your deploy\n- Project bootstrapping with optional Terraform support\n- Function metrics and cost analysis\n- Ignore deploying files with .apexignore\n- Function rollback support\n- Tail function logs\n- Concurrency for quick deploys\n- Dry-run to preview changes\n- VPC support\n- Multiple region support\n- Lambda@Edge support\n\n## Sponsors\n\nDoes your company use Apex? "functions" directory. Here\'s an example file structure:\n\n```\nproject.json\nfunctions\n\xe2\x94\x9c\xe2\x94\x80\xe2\x94\x80 bar\n\xe2\x94\x82\xc2\xa0\xc2\xa0 \xe2\x94\x9c\xe2\x94\x80\xe2\x94\x80 function.json\n\xe2\x94\x82\xc2\xa0\xc2\xa0 \xe2\x94\x94\xe2\x94\x80\xe2\x94\x80 index.js\n\xe2\x94\x94\xe2\x94\x80\xe2\x94\x80 foo\n    \xe2\x94\x9c\xe2\x94\x80\xe2\x94\x80 function.json\n    \xe2\x94\x94\xe2\x94\x80\xe2\x94\x80 index.js\n```\n\nThe `project.json` file defines project level configuration that applies to all functions, and defines dependencies. For this simple example the following will do:\n\n```json\n{\n  "name": "example",\n  "description": "Example project"\n}\n```\n\nEach function uses a `function.json` configuration file to define function-specific properties such as the runtime, amount of memory allocated, and timeout. This file is completely optional, as you can specify defaults in your `project.json` file. For example:\n\n```json\n{\n  "name": "bar",\n  "description": "Node.js example function",\n  "runtime": "nodejs4.3",\n  "memory": 128,\n  "timeout": 5,\n  "role": "arn:aws:iam::293503197324:role/lambda"\n}\n```\n\nNow the directory structure for your project would be:\n\n```\nproject.json\nfunctions\n\xe2\x94\x9c\xe2\x94\x80\xe2\x94\x80 bar\n\xe2\x94\x82\xc2\xa0\xc2\xa0 \xe2\x94\x94\xe2\x94\x80\xe2\x94\x80 index.js\n\xe2\x94\x94\xe2\x94\x80\xe2\x94\x80 foo\n    \xe2\x94\x94\xe2\x94\x80\xe2\x94\x80 index.js\n```\n\nFinally the source for the functions themselves look like this in Node.js:\n\n```js\nconsole.log(\'start bar\')\nexports.handle = function(e, ctx) {\n  ctx.succeed({ hello: e.name })\n}\n```\n\nApex operates at the project level, but many commands allow you to specify specific functions. For example you may deploy the entire project with a single command:\n\n```\n$ apex deploy\n```\n\nOr whitelist functions to deploy:\n\n```\n$ apex deploy foo bar\n```\n\nInvoke it!\n\n```\n$ echo \'{ "name": "Tobi" }\' | apex invoke bar\n{ "hello": "Tobi" }\n```\n\nSee the [Documentation](http://apex.run) for more information.\n\n## Links\n\n- [Website](http://apex.run)\n- [Medium](https://medium.com/@tjholowaychuk)\n- [Slack](https://chat.apex.sh/)\n\n---\n\n[![Build Status](https://semaphoreci.com/api/v1/projects/d27ff350-b9c5-4d99-96e5-64b1afb441c5/649392/badge.svg)](https://semaphoreci.com/tj/apex)\n[![Slack Status](https://apex-slackin.herokuapp.com/badge.svg)](https://apex-slackin.herokuapp.com/)\n[![GoDoc](https://godoc.org/github.com/apex/apex?status.svg)](https://godoc.org/github.com/apex/apex)\n![](https://img.shields.io/badge/license-MIT-blue.svg)\n![](https://img.shields.io/badge/status-stable-green.svg) [![OpenCollective](https://opencollective.com/apex/backers/badge.svg)](#backers) [![OpenCollective](https://opencollective.com/apex/sponsors/badge.svg)](#sponsors)\n\n<a href="https://apex.sh"><img src="http://tjholowaychuk.com:6000/svg/sponsor"></a>\n'
