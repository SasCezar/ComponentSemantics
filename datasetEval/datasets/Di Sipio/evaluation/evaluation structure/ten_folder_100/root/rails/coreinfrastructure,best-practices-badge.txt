plans)\n\n## Summary of Best Practices Criteria "passing" level\n\nThis is a summary of the passing criteria, with requirements in bold\n(for details, see the [full list of criteria](doc/criteria.md)):\n\n* **Have a [stable website](doc/criteria.md#homepage_url)**, which says:\n  - **[what it does](doc/criteria.md#description_good)**\n  - **[how to get it](doc/criteria.md#interact)**\n  - **[how to give feedback](doc/criteria.md#interact)**\n  - **[how to contribute](doc/criteria.md#contribution)** and\n    [preferred styles](doc/criteria.md#contribution_requirements)\n* **[Explicitly specify](doc/criteria.md#license_location) a\n  [FLOSS](doc/criteria.md#floss_license) [license](doc/criteria.md#floss_license_osi)**\n* **[Support HTTPS on the project sites](doc/criteria.md#sites_https)**\n* **[Document how to install and run (securely)](doc/criteria.md#documentation_basics),\n  and [any API](doc/criteria.md#documentation_interface)**\n* **Have a** [distributed](doc/criteria.md#repo_distributed)\n  **[public version control system](doc/criteria.md#repo_public),\n including [changes between releases](doc/criteria.md#repo_interim)**:\n  - **[Give each release a unique version](doc/criteria.md#version_unique)**, using\n    [semantic versioning format](doc/criteria.md#version_semver)\n  - **Give a [summary of changes for each release](doc/criteria.md#release_notes),\n    [identifying any fixed vulnerabilities](doc/criteria.md#release_notes_vulns)**\n* **Allow [bug reports to be submitted](doc/criteria.md#report_process),\n  [archived](doc/criteria.md#report_archive)** and\n  [tracked](doc/criteria.md#report_tracker):\n  - **[Acknowledge](doc/criteria.md#report_responses)**/respond to bugs &\n    [enhancement requests](doc/criteria.md#enhancement_responses), rather than\n    ignoring them\n  - **Have a [secure](doc/criteria.md#vulnerability_report_private),\n    [documented process](doc/criteria.md#vulnerability_report_process) for\n    reporting vulnerabilities**\n  - **[Respond within 14 days](doc/criteria.md#vulnerability_report_response),\n    and [fix vulnerabilities](doc/criteria.md#vulnerabilities_critical_fixed),\n    [within 60 days if they\'re public](doc/criteria.md#vulnerabilities_fixed_60_days)**\n* **[Have a build that works](doc/criteria.md#build)**, using\n  [standard](doc/criteria.md#build_common_tools)\n  [open-source](doc/criteria.md#build_floss_tools) tools\n  - **Enable (and [fix](doc/criteria.md#warnings_fixed))\n    [compiler warnings and lint-like checks](doc/criteria.md#warnings)**\n  - **[Run tools](doc/criteria.md#dynamic_analysis)\n    ([sanitizers/Valgrind](doc/criteria.md#dynamic_analysis_unsafe) etc.)\n  - [Run a fuzzer or web-scanner over the code](doc/criteria.md#dynamic_analysis)\n* **[Have a developer who understands secure software](doc/criteria.md#know_secure_design)\n  and [common vulnerability errors](doc/criteria.md#know_common_errors)**\n* If cryptography is used:\n  - **[Use public protocols/algorithm](doc/criteria.md#crypto_published)**\n  - **[Don\'t re-implement standard  - **[Don\'t use known-broken](doc/criteria.md#crypto_working)** or\n    [known-weak](doc/criteria.md#crypto_weaknesses) algorithms\n  - [Use algorithms with forward of pursuing projects have a passing badge.\nThat said, some projects would like to meet even stronger criteria,\nand many users would like projects to do so.\nWe have established two higher levels beyond passing: silver and gold.\nThe higher levels strengthen some of the passing criteria and add new\ncriteria of their own.\n\n### Silver\n\nHere is a summary of the silver criteria, with requirements in bold\n(for details, see the [full list of silver criteria](doc/other.md)):\n\n* **[Use a DCO or similar](doc/other.md#dco)**\n* **[Define/document project governance](doc/other.md#governance)**\n* **[Another will have the necessary access rights if someone dies](doc/other.md#access_continuity)**\n* *["Bus factor" of 2 or more](doc/other.md#bus_factor)*\n* **[Document security requirements](doc/other.md#security_requirements)**\n* **[Have an assurance case explaining why security requirements are met](doc/other.md#assurance_case)**\n* **[Have a quick start guide](doc/other.md#documentation_quick_start)**\n* *[Follow accessibility best practices](doc/other.md#accessibility_best_practices)*\n* **[Pick & follow coding standards](doc/other.md#coding_standards)**\n* **[Monitor external dependencies to detect/fix known vulnerabilities](doc/other.md#dependency_monitoring)**\n* **[Tests have 80%+ statement coverage](doc/other.md#test_statement_coverage80)**\n* **[Project releases for widespread use are cryptographically signed](doc/other.md#signed_releases)**\n* **[Check all inputs from potentially untrusted sources for validity (using a whitelist)](doc/other.md#input_validation)**\n* *[Use hardening mechanisms](doc/other.md#hardening)*\n\n### Gold\n\nHere is a summary of the gold criteria, with requirements in bold\n(for details, see the [full list of gold criteria](doc/other.md)):\n\n* **[At least 2 unassociated significant contributors](doc/other.md#contributors_unassociated)**\n* **[Per-file copyright and license](doc/other.md#copyright_per_file)**\n* **[Use 2FA](doc/other.md#require_2FA)**\n* **[At least 50% of all modifications are reviewed by another](doc/other.md#two_person_review)**\n* **[Have a reproducible build](doc/other.md#reproducible_build)**\n* **[Use continuous integration](doc/other.md#test_continuous_integration)**\n* **[Statement coverage 90%+](doc/other.md#test_statement_coverage90)**\n* **[Branch coverage 80%+](doc/other.md#test_branch_coverage80)**\n* **[Support secure protocols & disable insecure protocols by default](doc/other.md#crypto_used_network)**\n* **[Use TLS version 1.2 or higher](doc/other.md#crypto_tls12)**\n* **[Have a hardened project website, repo, and download site](doc/other.md#hardened_site)**\n* **[Have a security review (internal or external)](doc/other.md#security_review)**\n\n## License\n\nAll material here is released
