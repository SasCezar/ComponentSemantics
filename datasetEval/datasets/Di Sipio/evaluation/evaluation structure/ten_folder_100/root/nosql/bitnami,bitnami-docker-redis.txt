To avoid this loss of data, you should mount a volume that will persist even after the container is removed.\n\nFor persistence you should mount a directory at the `/bitnami` path. If the mounted directory is empty, it will be initialized on the first run.\n\n\n```bash\n$ docker run \\\n    -e ALLOW_EMPTY_PASSWORD=yes \\\n    -v /path/to/redis-persistence:/bitnami/redis/data \\\n    bitnami/redis:latest\n```\n\nYou can also do this by modifying the [`docker-compose.yml`](https://github.com/bitnami/bitnami-docker-redis/blob/master/docker-compose.yml) file present in this repository:\n\n```yaml\nservices:\n  redis:\n  ...\n    volumes:\n      - /path/to/redis-persistence:/bitnami/redis/data\n  ...\n```\n\n> NOTE: As this is a non-root container, the mounted files and directories must have the proper permissions for the UID `1001`.\n\n# Connecting to other containers\n\nUsing [Docker container networking](https://docs.docker.com/engine/userguide/networking/), a Redis server running inside a container can easily be accessed by your application containers.\n\nContainers attached to the same network can communicate with each other using the container name as the hostname.\n\n## Using the Command Line\n\nIn this example, we will create a Redis client instance that will connect to the server instance that is running on the same docker network as the client.\n\n### Step 1: Create a network\n\n```bash\n$ docker network create app-tier --driver bridge\n```\n\n### Step 2: Launch the Redis server instance\n\nUse the `--network app-tier` argument to the `docker run` command to attach the Redis container to the `app-tier` network.\n\n```bash\n$ docker run -d --name redis-server \\\n    -e ALLOW_EMPTY_PASSWORD=yes \\\n    --network app-tier \\\n    bitnami/redis:latest\n```\n\n### Step 3: Launch your Redis client instance\n\nFinally we create a new container instance to launch the Redis client and connect to the server created in the previous step:\n\n```bash\n$ docker run -it --rm \\\n    --network app-tier \\\n    bitnami/redis:latest redis-cli -h redis-server\n```\n\n## Using Docker Compose\n\nWhen not specified, Docker Compose automatically sets up a new network and attaches all deployed services to that network. However, we will explicitly define a new `bridge` network named `app-tier`. In this example we assume that you want to connect to the Redis server from your own custom application image which is identified in the following snippet by the service name `myapp`.\n\n```yaml\nversion: \'2\'\n\nnetworks:\n  app-tier:\n    driver: bridge\n\nservices:\n  redis:\n    image: \'bitnami/redis:latest\'\n    environment:\n      - ALLOW_EMPTY_PASSWORD=yes\n    networks:\n      - app-tier\n  myapp:\n    image: \'YOUR_APPLICATION_IMAGE\'\n    networks:\n      - app-tier\n```\n\n> **IMPORTANT**:\n>\n> 1. Please update the **YOUR_APPLICATION_IMAGE_** placeholder in the above snippet with your application image\n> 2. In your application container, use the hostname `redis` to connect to the Redis server\n\nLaunch the containers using:\n\n```bash\n$ docker-compose up -d\n```\n\n# Configuration\n\n## Disabling Redis commands\n\nFor security reasons, you may want to disable some commands. You can specify them by using the following environment variable on the first run:\n\n- `DISABLE_COMMANDS`: Comma-separated list of Redis commands to disable. Defaults to empty.\n\n```bash\n$ docker run --name redis -e DISABLE_COMMANDS=FLUSHDB,FLUSHALL,CONFIG bitnami/redis:latest\n```\n\nAlternatively, modify the [`docker-compose.yml`](https://github.com/bitnami/bitnami-docker-redis/blob/master/docker-compose.yml) file present in this repository:\n\n```yaml\nservices:\n  redis:\n  ...\n    environment:\n      - DISABLE_COMMANDS=FLUSHDB,FLUSHALL,CONFIG\n  ...\n```\n\nAs specified in the docker-compose, `FLUSHDB` and `FLUSHALL` commands are disabled. Comment out or remove the\nenvironment variable if you don\'t want to disable any commands:\n\n```yaml\nservices:\n  redis:\n  ...\n    environment:\n      # - DISABLE_COMMANDS=FLUSHDB,FLUSHALL\n  ...\n```\n\n## Passing extra command-line flags to redis-server startup\n\nPassing extra command-line flags to the redis service command is possible by adding them as arguments to *run.sh* script:\n\n```bash\n$ docker run --name redis -e ALLOW_EMPTY_PASSWORD=yes bitnami/redis:latest /run.sh --maxmemory 100mb\n```\n\nAlternatively, modify the [`docker-compose.yml`](https://github.com/bitnami/bitnami-docker-redis/blob/master/docker-compose.yml) file present in this repository:\n\n```yaml\nservices:\n  redis:\n  ...\n    environment:\n      - ALLOW_EMPTY_PASSWORD=yes\n    command: /run.sh --maxmemory 100mb\n  ...\n```\n\nRefer to the [Redis documentation](https://redis.io/topics/config#passing-arguments-via-the-command-line) for the complete list of arguments.\n\n## Setting the server password on first run\n\nPassing the `REDIS_PASSWORD` environment variable when running the image for the first time will set the Redis server password to the value of `REDIS_PASSWORD` (or the content of the file specified in `REDIS_PASSWORD_FILE`).\n\n```bash\n$ docker run --name redis -e REDIS_PASSWORD=password123 bitnami/redis:latest\n```\n\nAlternatively, modify the [`docker-compose.yml`](https://github.com/bitnami/bitnami-docker-redis/blob/master/docker-compose.yml) file present in this repository:\n\n```yaml\nservices:\n  redis:\n  ...\n    environment:\n      - REDIS_PASSWORD=password123\n  ...\n```\n\n**Warning** The Redis database is always configured with remote access enabled. It\'s suggested that the `REDIS_PASSWORD` env variable is always specified to set a password. In case you want to access the database without a password set the environment variable `ALLOW_EMPTY_PASSWORD=yes`. **This is recommended only for development**.\n\n## Allowing empty passwords\n\nBy default the Redis image expects all the available passwords to be set. In order to allow empty passwords, it is necessary to set the `ALLOW_EMPTY_PASSWORD=yes` env variable. This env variable is only recommended for testing or development purposes. We strongly recommend specifying the `REDIS_PASSWORD` for any other scenario.\n\n```bash\n$ docker run --name redis -e ALLOW_EMPTY_PASSWORD=yes bitnami/redis:latest\n```\n\nAlternatively, modify the [`docker-compose.yml`](https://github.com/bitnami/bitnami-docker-redis/blob/master/docker-compose.yml) file present in this repository:\n\n```yaml\nservices:\n  redis:\n  ...\n    environment:\n      - ALLOW_EMPTY_PASSWORD=yes\n  ...\n```\n\n## Setting up replication\n\nA [replication](http://redis.io/topics/replication) cluster can easily be setup with the Bitnami Redis Docker Image using the following environment variables:\n\n - `REDIS_REPLICATION_MODE`: The replication mode. Possible values `master`/`slave`. No defaults.\n - `REDIS_MASTER_HOST`: Hostname/IP of replication master (replica node parameter). No defaults.\n - `REDIS_MASTER_PORT_NUMBER`: Server port of the replication master (replica node parameter). Defaults to `6379`.\n - `REDIS_MASTER_PASSWORD`: Password to authenticate with the master (replica node parameter). No defaults. As an alternative, you can mount a file with the password and set the `REDIS_MASTER_PASSWORD_FILE` variable.\n\nIn a replication cluster you can have one master and zero or more replicas. When replication is enabled the master node is in read-write mode, while the replicas are in read-only mode. For best performance its advisable to limit the reads to the replicas.\n\n### Step 1: Create the replication master\n\nThe first step is to start the Redis master.\n\n```bash\n$ docker run --name redis-master \\\n  -e REDIS_REPLICATION_MODE=master \\\n  -e REDIS_PASSWORD=masterpassword123 \\\n  bitnami/redis:latest\n```\n\nIn the above command the container is configured as the `master` using the `REDIS_REPLICATION_MODE` parameter. The `REDIS_PASSWORD` parameter enables authentication on the Redis master.\n\n### Step 2: Create the replica node\n\nNext we start a Redis replica container.\n\n```bash\n$ docker run --name redis-replica \\\n  --link redis-master:master \\\n  -e REDIS_REPLICATION_MODE=slave \\\n  -e REDIS_MASTER_HOST=master \\\n  -e REDIS_MASTER_PORT_NUMBER=6379 \\\n  -e REDIS_MASTER_PASSWORD=masterpassword123 \\\n  -e REDIS_PASSWORD=password123 \\\n  bitnami/redis:latest\n```\n\nIn the above command the container is configured as a `slave` using the `REDIS_REPLICATION_MODE` parameter. The `REDIS_MASTER_HOST`, `REDIS_MASTER_PORT_NUMBER` and `REDIS_MASTER_PASSWORD ` parameters are used connect and authenticate with the Redis master. The `REDIS_PASSWORD` parameter enables authentication on the Redis replica.\n\nYou now have a two node Redis master/replica replication cluster up and running which can be scaled by adding/removing replicas.\n\nIf the Redis master goes down you can reconfigure a replica to become a master using:\n\n```bash\n$ docker exec redis-replica redis-cli -a password123 SLAVEOF NO ONE\n```\n\n> **Note**: The configuration of the other replicas in the cluster needs to be updated so that they are aware of the new master. In our example, this would involve restarting the other replicas with `--link redis-replica:master`.\n\nWith Docker Compose the master/replica mode can be setup using:\n\n```yaml\nversion: \'2\'\n\nservices:\n  redis-master:\n    image: \'bitnami/redis:latest\'\n    ports:\n      - \'6379\'\n    environment:\n      - REDIS_REPLICATION_MODE=master\n      - REDIS_PASSWORD=my_master_password\n    volumes:\n      - \'/path/to/redis-persistence:/bitnami\'\n\n  redis-replica:\n    image: \'bitnami/redis:latest\'\n    ports:\n      - \'6379\'\n    depends_on:\n      - redis-master\n    environment:\n      - REDIS_REPLICATION_MODE=slave\n      - REDIS_MASTER_HOST=redis-master\n      - REDIS_MASTER_PORT_NUMBER=6379\n      - REDIS_MASTER_PASSWORD=my_master_password\n      - REDIS_PASSWORD=my_replica_password\n```\n\nScale the number of replicas using:\n\n```bash\n$ docker-compose up --detach --scale redis-master=1 --scale redis-secondary=3\n```\n\nThe above command scales up the number of replicas to `3`. You can scale down in the same way.\n\n> **Note**: You should not scale up/down the number of master nodes. Always have only one master node running.\n\n## Configuration file\n\nThe image looks for configurations in `/opt/bitnami/redis/etc/redis.conf`. You can overwrite the `redis.conf` file using your own custom configuration file.\n\n```bash\n$ docker run --name redis \\\n    -e ALLOW_EMPTY_PASSWORD=yes \\\n    -v /path/to/your_redis.conf:/opt/bitnami/redis/etc/redis.conf \\\n    -v /path/to/redis-data-persistence:/bitnami/redis/data \\\n    bitnami/redis:latest\n```\n\nAlternatively, modify the [`docker-compose.yml`](https://github.com/bitnami/bitnami-docker-redis/blob/master/docker-compose.yml) file present in this repository:\n\n```yaml\nservices:\n  redis:\n  ...\n    volumes:\n      - /path/to/your_redis.conf:/opt/bitnami/redis/etc/redis.conf\n      - /path/to/redis-persistence:/bitnami/redis/data\n  ...\n```\n\nRefer to the [Redis configuration](http://redis.io/topics/config) manual for the complete list of configuration options.\n\n# Logging\n\nThe Bitnami Redis Docker image sends the container logs to the `stdout`. To view the logs:\n\n```bash\n$ docker logs redis\n```\n\nor using Docker Compose:\n\n```bash\n$ docker-compose logs redis\n```\n\nYou can configure the containers [logging driver](https://docs.docker.com/engine/admin/logging/overview/) using the `--log-driver` option if you wish to consume the container logs differently. For backward compatibility with previous versions, `slave` replication mode is still supported. We encourage the use of the `REPLICAOF` command if you are using Redis 5.0.\n\n## 4.0.1-r24\n\n- Decrease the size of the container. It is not necessary Node.js anymore. Redis configuration moved to bash scripts in the `rootfs/` folder.\n- The recommended mount point to persist data changes to `/bitnami/redis/data`.\n- The main `redis.conf` file is not persisted in a volume. The path is `/opt/bitnami/redis/etc/redis.conf`.\n- Backwards compatibility is not guaranteed when data is persisted using docker-compose. You can use the workaround below to overcome it:\n\n```bash\ndocker-compose down\n# Locate your volume and modify the file tree\nVOLUME=$(docker volume ls | grep "redis_data" | awk \'{print $2}\')\ndocker run --rm -i -v=${VOLUME}:/tmp/redis busybox find /tmp/redis/data -maxdepth 1 -exec mv {} /tmp/redis \\;\ndocker run --rm -i -v=${VOLUME}:/tmp/redis busybox rm -rf /tmp/redis/{data,conf,.initialized}\n# Change the mount point\nsed -i -e \'s#redis_data:/bitnami/redis#redis_data:/bitnami/redis/data#g\' docker-compose.yml\n# Pull the latest bitnami/redis image\ndocker pull bitnami/redis:latest\ndocker-compose up -d\n```\n\n## 4.0.1-r1\n\n- The redis container has been migrated to a non-root container approach. Previously the container run as `root` user and the redis daemon was started as `redis` user. From now own, both the container and the redis daemon run as user `1001`.\n  As a consequence, the configuration files are writable by the user running the redis process.\n\n## 3.2.0-r0\n\n- All volumes have been merged at `/bitnami/redis`. Now you only need to mount a single volume at `/bitnami/redis` for persistence.\n- The logs are always sent to the `stdout` and are no longer collected in the volume.\n\n# Contributing\n\nWe\'d love for you to contribute to this container.
