Reactive gateway based on webflux\n\n# Architecture\n \n ![](https://yu199195.github.io/images/soul/soul-framework.png)  \n  \n# Execution Flow\n \n ![](https://yu199195.github.io/images/soul/soul-handler.png)\n  \n# Modules\n\n * soul-admin : Plug-in and other information configuration management background\n \n * soul-bootstrap : With the startup project, users can refer to\n \n * soul-common :  Framework common class\n \n * soul-configuration : zookeeper configuration project\n \n * soul-spring-boot-starter : Support for the spring boot starter\n \n * soul-web : Core processing packages include plug-ins, request routing and forwarding, and so on\n \n * soul-extend-demo : Demo of the extension point\n \n * soul-test : the rpc test project\n\n# Features\n\n   * It provides plugins such as current limiting, fusing, forwarding, routing monitoring and so on.\n   \n   * Seamless docking with HTTP,Restful,websocket,dubbo and springcloud.\n   \n   * Plug-in hot plug, users can customize the development.\n   \n   * Selectors and rules are dynamically configured for flexible matching.\n\n   * Support for cluster deployment.\n   \n   * Support A/B test and grayscale publishing\xe3\x80\x82\n   \n\n# Plugin\n\n Whenever a request comes in ,Soul Execute all open plug-ins through the chain of responsibility.\n \n Plugins are the heart of soul And plug-ins are extensible and hot-pluggable.\n \n Different plug-ins do different things \n \n Of course, users can also customize plug-ins to meet their own needs.\n \n If you want to customize, see [plugin-extend](https://dromara.org/website/zh-cn/docs/soul/extend.html)\n \n\n# Selector & rule \n\n  According to your HTTP request headers, selectors and rules are used to route your requests.\n  \n  Selector is your first route, It is coarser grained, for example, at the module level.\n  \n  Rule is your second route and what do you think your request should do,For example a method level in a module.\n  \n  The selector and the rule match only once, and the match is returned. So the coarsest granularity should be sorted last.\n   \n  \n# Data Caching  & Data Sync\n \n  All data is cached ConcurrentHashMap in the JVM So it\'s very fast.\n  \n  When the user is managing changes in the background,\n  \n  Soul dynamically updates the cache by listening to the zookeeper node, websocket push,http longPull.\n  \n  ![Data Sync](https://bestkobe.gitee.io/images/soul/soul-config-processor.png?_t=201908032316)\n  \n  ![Sync Flow](https://bestkobe.gitee.io/images/soul/config-strage-processor.png?_t=201908032339)\n \n#
