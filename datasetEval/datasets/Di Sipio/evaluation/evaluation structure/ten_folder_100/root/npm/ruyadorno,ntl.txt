/>\n\n## :mag_right: Usage\n\nNavigate to any folder containing a `package.json` file (usually a Node.js project) that has configured **scripts**, then just use the ntl command:\n\n```sh\nntl\n```\n\nYou can also specify a path to a project folder containing a `package.json` file:\n\n```sh\nntl ./my-node-project\n```\n\n<br />\n<br />\n\n<p align="center">\n<img alt="demo animation" width="540" src="https://ruyadorno.github.io/svg-demos/ntl/examples/intro.svg" />\n</p>\n\n<br />\n\n## :heart_eyes: Features\n\n- Interactive interface that lists all `package.json` scripts\n- Select any item in the interactive interface to execute that task\n- Add descriptions to each task that can be shown in the UI\n- Multiple interactive interfaces (menu list, autocomplete fuzzy search)\n- Many options to customize the UI (exclude scripts, amount of items, etc)\n- Easy to repeat last ran script (`nt` and `rerun` options)\n- Run multiple tasks (can also easily repeat multiple ran tasks)\n- Customize rerun cache options\n\n<br />\n\n## :books: Customize\n\n### Custom runner\n\nBy default **Node Task List** tries to use [npm](https://github.com/npm/cli/) to run the configured `script` tasks. In case you want to use a custom runner (e.g: `yarn` or `pnpm`) you can configure **ntl** to use whatever runner you prefer.\n\nThat can be configured system-wide by setting the environment variable in your profile file (`.bashrc` or `.bash_profile` for macos). In case you only want to ever use **yarn** as your node task runner, you should set:\n\n```sh\n# .bashrc (linux) OR .bash_profile (macos)\nexport NTL_RUNNER=yarn\n```\n\nKeep in mind environment variables are flexible enough that they can also be set temporarily prior to running a command, so the following would also work:\n\n```sh\nNTL_RUNNER=yarn ntl\n```\n\n:heavy_exclamation_mark: You can also define a `runner` in a per-project basis using the `ntl` configuration of your `package.json`, e.g:\n\n```json\n{\n  "name": "<project>",\n  "version": "1.0.0",\n  "ntl": {\n    "runner": "yarn"\n  }\n}\n```\n\n<br />\n\n### Using task descriptions\n\n:memo: You can define descriptions for your tasks in your `package.json` file by defining a `ntl` section, e.g:\n\n```json\n{\n  "name": "<project>",\n  "version": "1.0.0",\n  "scripts": {\n    "build": "make build",\n    "coverage": "jest --coverage",\n    "test": "jest"\n  },\n  "ntl": {\n    "descriptions": {\n      "build": "Builds the project",\n      "coverage": "Run test outputing code coverage",\n      "test": "Run project\'s tests"\n    }\n  }\n}\n```\n\nThese descriptions will be shown anytime you run `ntl`.\n\n<br />\n\n### Displaying task contents\n\nUse the `--info` or simply `-i` option in order to display the contents of each script task, like:\n\n```sh\n$ ntl -i\n\xe2\xac\xa2  Node Task List\n? Select a task to run: (Use arrow keys)\n\xe2\x9d\xaf generate-manual \xe2\x80\xba maked-man README.md > man/man1/ntl.1\n          pretest \xe2\x80\xba eslint cli.js rerun.js test\n             test \xe2\x80\xba cross-env NTL_NO_RERUN_CACHE=1 tap\n```\n\nTask contents will also be shown when using the `--descriptions` option and no description is available for a given item.\n\n<br />\n\n### Exclude tasks from UI\n\nYou can define a list of scripts to be excluded from the interactive menu using the `--exclude` option:\n\n```\n$ ntl -e coverall tasks\n\xe2\xac\xa2  Node Task List\n? Select a task to run: (Use arrow keys)\n\xe2\x9d\xaf test\n  test:watch\n```\n\nYou can also use a wildcard character to exclude multiple scripts:\n\n```\n$ ntl -e "test*"\n\xe2\xac\xa2  Node Task List\n? Select a task to run: (Use arrow keys)\n\xe2\x9d\xaf coveralls\n  tasks\n```\n\n<br />\n\n### Exclude tasks with missing descriptions\n\nYou can also filter out items that doesn\'t have a description using the `--descriptions-only` or `-o` option.\n\n<br />\n\n### Customize cache\n\n**ntl** uses a cache system that stores the last ran command for each project in order to make it easier for users to repeat it. Given its importance, the following environment variables are available in order to customize its location and size:\n\n- `NTL_RERUN_CACHE_DIR`: Defines a directory to store the cache file\n- `NTL_RERUN_CACHE_NAME`: Filename to use for the cache\n- `NTL_RERUN_CACHE_MAX`: Number of items to store in the cache (defaults to `10`)\n- `NTL_NO_RERUN_CACHE`: When defined, avoid the cache system completely\n\nFor example, if a given user wanted to store its cache in `~/.ntl/cache` location and save up to 100 items in it, they could add the following to their `.bashrc` (linux) or `.bash_profile` (macos):\n\n```sh\nexport NTL_RERUN_CACHE_DIR=$HOME\nexport NTL_RERUN_CACHE_NAME=cache\nexport NTL_RERUN_CACHE_MAX=100\n```\n\nThe cache can also be customized through command line options:\n\n- `--rerun-cache-dir` Defines a directory to store the cache file\n- `--rerun-cache-name`: Filename to use for the cache\n- `--no-rerun-cache`: Avoids the cache system completely\n\n<br />\n\n### UI Size\n\nYou can increase/reduce the size of the presented UI list using the `--size` or `-s` option. In this example we just increased the size of the list to show up to 12 items at once:\n\n```\n$ ntl -s 12\n\xe2\xac\xa2  Node Task List\n? Select a task to run:\n\xe2\x9d\xaf build\n  generate-manual\n  hello\n  bomdia\n  bonjour\n  test\n  test:dev\n  test:ci\n  test:integration\n  test:unit\n  test:e2e\n  start\n```\n\nThe default size value is 7 items.\n\n<br />\n\n## :arrows_counterclockwise: Repeat the last ran task\n\n**ntl** provides many options to make it easier to rerun the last task, either through having it selected as default option the next time you run the `ntl` command, or by using one of the following:\n\n- **The ultra convenient way**: `nt` command shorthand :sunglasses: (You should think of `nt` as: "ok, just run the last node task", in contrast to `ntl` which should be interpreted as: "ok, give me the node task list again") in case no previous task is available, running `nt` will behave exactly as `ntl`\n- Using a `--rerun` or `-r` flag, e.g: `ntl -r`\n- Prepending the `NTL_RERUN` env variable, e.g: `NTL_RERUN=true ntl`\n\n<p align="center">\n<img alt="demo animation showing how to rerun a script using the nt command shorthand" width="540" src="https://ruyadorno.github.io/svg-demos/ntl/examples/option-rerun.svg" />\n</p>\n\n<br />\n\n## :m: Run multiple tasks\n\nUsing the `--multiple` or `-m` option, the interface becomes a checkbox-based list that allows you to select multiple tasks and run them in serial.\n\n```sh\n$ ntl -m\n\xe2\xac\xa2  Node Task List\n? Select a task to run:\n \xe2\x97\xaf start\n \xe2\x97\x89 test\n\xe2\x9d\xaf\xe2\x97\x89 test:ci\n```\n\n<p align="center">\n<img alt="demo animation showing how to use -m to have a checkbox interface allowing for selection of multiple tasks to be ran in serial" width="540" src="https://ruyadorno.github.io/svg-demos/ntl/examples/option-multiple.svg" />\n</p>\n\nBetter yet, combine that with the **rerun** feature and you can repeat multiple tasks using the `nt` command.\n\n<br />\n\n## :fast_forward: Run in autocomplete or fuzzy search mode\n\nUse `--autocomplete` or `-A` option in order to use an interface variation that allows you to type the name of the task instead of browsing through an arrow-based menu. This mode can be very helpful when managing a long list of tasks.\n\n```sh\n$ ntl -A\n\xe2\xac\xa2  Node Task List\n? Select a task to run: t\n\xe2\x9d\xaf pretest\n  test\n  start\n```\n\n<p align="center">\n<img alt="demo animation showing how to use -m to have a checkbox interface allowing for selection of multiple tasks to be ran in serial" width="540" src="https://ruyadorno.github.io/svg-demos/ntl/examples/option-autocomplete.svg" />\n</p>\n\n<br />\n\n## :white_check_mark: Tips\n\n### ntl as default task\n\nYou can define `ntl` as a dev dependency and one of the tasks of your project, specially `start` - so whenever someone runs `npm start` or `yarn start` they get the convenient **ntl** interface. Like in the following `package.json` example:\n\n```json\n{\n  "name": "<project>",\n  "version": "1.0.0",\n  "scripts": {\n    "start": "ntl"\n  },\n  "devDependencies": {\n    "ntl": "^5.0.0"\n  }\n}\n```\n\n### Exclude scripts\n\nYou can also define a task that invokes `ntl` while excluding other tasks, e.g:\n\n```json\n{\n  "scripts": {\n    "test": "jest --coverage",\n    "test:watch": "jest --coverage --watchAll",\n    "coveralls": "jest --coverage --coverageReporters=text-lcov | coveralls",\n    "tasks": "ntl --exclude coverall tasks"\n  }\n}\n```\n\n### Included command aliases\n\n- `ntl` The default command\n- `nodetasklist` Longhand version in case users have conflicting `ntl` commands\n- `npm-tasklist` Legacy longhand version\n- `nt` Rerun last script shortcut\n- `nodetask` Rerun last script longhand\n\n\n<br />\n\n## :information_source: Help\n\nStill feel like you could use some `--help` ?\n\n```sh\nUsage:\n  ntl [<path>]             Build an interactive
