(based on MySQL syntax), resolves and optimizes queries.\nIt provides simple interfaces to allow custom tabular data source implementations.\n\n**go-mysql-server** also provides a server implementation compatible with the MySQL wire protocol.\nThat means it is compatible with MySQL ODBC, JDBC, or the default MySQL client shell interface.\n\n## Scope of this project\n\nThese are the goals of **go-mysql-server**:\n\n- Be a generic extensible SQL engine that performs queries on your data sources.\n- Provide interfaces so you can implement your own custom data sources without providing any (except for the `mem` data source that is used for testing purposes).\n- Have a runnable server you can use on your specific implementation.\n- Parse and optimize queries while still allow specific implementations to add their own analysis steps and optimizations.\n- Provide some common index driver implementations so the user does not have to bring their own index implementation, and still be able to do so if they need to.\n\nWhat are not the goals of **go-mysql-server**:\n\n- Be a drop-in MySQL database replacement.\n- Be an application/server you can use directly.\n- Provide any kind of backend implementation (other than the `mem` one used for testing) such as json, csv, yaml, ... That\'s for clients to implement and use.\n\nWhat\'s the use case of **go-mysql-server**?\n\nHaving data in another format that you want as tabular data to query using SQL, such as git. As an example of this, we have [gitbase](https://github.com/src-d/gitbase).\n\n## Installation\n\nThe import path for the package is `github.com/src-d/go-mysql-server`.\n\nTo install it, run:\n\n```\ngo get github.com/src-d/go-mysql-server\n```\n\n## Documentation\n\n* [go-mysql-server godoc](https://godoc.org/github.com/src-d/go-mysql-server)\n\n\n## SQL syntax\n\nWe are continuously adding more functionality to go-mysql-server. So, if you want to start a server, first instantiate the engine and pass your `sql.Database` implementation.\n\nIt will be in charge of handling all the logic to retrieve the data from your source.\nHere you can see an example using the in-memory database implementation:\n\n```go\n...\n\nfunc main() {\n    driver := sqle.NewDefault()\n    driver.AddDatabase(createTestDatabase())\n\n    config := server.Config{\n        Protocol: "tcp",\n        Address:  "localhost:3306",\n        Auth:     auth.NewNativeSingle("user", "pass", auth.AllPermissions),\n    }\n\n    s, err := server.NewDefaultServer(config, driver)\n    if err != nil {\n        panic(err)\n    }\n\n    s.Start()\n}\n\nfunc createTestDatabase() *memory.Database {\n    const (\n        dbName    = "test"\n        tableName = "mytable"\n    )\n\n    db := memory.NewDatabase(dbName)\n    table := memory.NewTable(tableName, sql.Schema{\n        {Name: "name", Type: sql.Text, Nullable: false, Source: tableName},\n        {Name: "email", Type: sql.Text, Nullable: false, Source: tableName},\n        {Name: "phone_numbers", Type: sql.JSON, Nullable: false, Source: tableName},\n        {Name: "created_at", Type: sql.Timestamp, Nullable: false, Source: tableName},\n    })\n\n    db.AddTable(tableName, table)\n    ctx := sql.NewEmptyContext()\n\n    rows := []sql.Row{\n        sql.NewRow("John Doe", "john@doe.com", []string{"555-555-555"}, time.Now()),\n        sql.NewRow("John Doe", "johnalt@doe.com", []string{}, time.Now()),\n        sql.NewRow("Jane Doe", "jane@doe.com", []string{}, time.Now()),\n        sql.NewRow("Evil Bob", "evilbob@gmail.com", []string{"555-666-555", "666-666-666"}, time.Now()),\n\t}\n\n    for _, row := range rows {\n        table.Insert(ctx, row)\n    }\n\n    return db\n}\n\n...\n```\n\nThen, you can connect to the server with any MySQL client:\n```bash\n> mysql --host=127.0.0.1 --port=3306 -u user -ppass test -e "SELECT * FROM interface. This interface will provide tables from your data source.\n  - If your database implementation supports adding more tables, you might want to add support for `sql.Alterable` interface\n\n- `sql.Table` interface. It will be in charge of transforming any kind of data into an iterator of Rows. Depending on how much you want to optimize the queries, you also can implement other interfaces on your tables:\n  - `sql.PushdownProjectionTable` interface will provide a way to get only the columns needed for the executed query.\n  - `sql.PushdownProjectionAndFiltersTable` interface will provide the same functionality described before, but also will push down the filters used in the executed query. It allows to filter data in advance, and speed up queries.\n  - `sql.Indexable` add index capabilities to your table. By implementing this interface you can create and use indexes on this table.\n  - `sql.Inserter` can be implemented if your data source tables allow insertions.\n\n- If you need some custom tree modifications, you can also implement your own `analyzer.Rules`.\n\nYou can see a really simple data source implementation on our `mem` package.\n\n## Indexes\n\n`go-mysql-server` exposes a series of interfaces to allow you to implement your own indexes so you can speedup your queries.\n\nTaking a look at the main [index interface](https://github.com/src-d/go-mysql-server/blob/master/sql/index.go#L35), you must note a couple of constraints:\n\n- This abstraction lets you create an index for multiple columns (one or more) or for **only one** expression (e.g. function applied on multiple columns).\n- If you want to index an expression that is not a column you will only be able to index **one and only one** expression at a time.\n\n## Custom index driver implementation\n\nIndex drivers provide different backends for storing and querying indexes. To implement a custom index driver you need to implement a few things:\n\n- `sql.IndexDriver` interface, which will be the driver itself. Not that your driver must return an unique ID in the `ID` method. This ID is unique for your driver and should not clash with any other registered driver. It\'s the driver\'s responsibility to be fault tolerant and be able to automatically detect and recover from corruption in indexes.\n- `sql.Index` interface, returned by your driver when an index is loaded or created.\n  - Your `sql.Index` may optionally implement the `sql.AscendIndex` and/or `sql.DescendIndex` interfaces, if you want to support more comparison operators like `>`, `<`, `>=`, `<=` or `BETWEEN`.\n- `sql.IndexLookup` interface, returned by your index in any of the implemented operations to get a subset of the indexed values.\n  - Your `sql.IndexLookup` may optionally implement the `sql.Mergeable` and `sql.SetOperations` interfaces if you want to support set operations to merge your index lookups.\n- `sql.IndexValueIter` interface, which will be returned by your `sql.IndexLookup` and should return the values of the index.\n- Don\'t forget to register the index driver in your `sql.Catalog` using `catalog.RegisterIndexDriver(mydriver)` to be able to use it.\n\nTo create indexes using your custom index driver you need to use `USING driverid` on the index creation query. For example:\n\n```sql\nCREATE INDEX foo ON table USING driverid (col1, col2)\n```\n\nYou can see an example of a driver implementation inside the `sql/index/pilosa` package, where the pilosa driver is implemented.\n\nIndex creation is synchronous by default, to make it asynchronous, use `WITH (async = true)`, for example:\n\n```sql\nCREATE INDEX foo ON table USING driverid (col1, col2) WITH (async = true)\n```\n\n### Old `pilosalib` driver\n\n`pilosalib` driver was renamed to `pilosa` and now `pilosa` does not require an external pilosa server. `pilosa` is not supported on Windows.\n\n### Metrics\n\n`go-mysql-server` utilizes `github.com/go-kit/kit/metrics` module to expose metrics (counters, gauges, histograms) for certain packages (so far for `engine`, `analyzer`, `regex`, `pilosa`). If you already have metrics server (prometheus, statsd/statsite, influxdb, etc.) and you want to gather metrics also from `go-mysql-server` components, you will need to initialize some global variables by particular implementations to satisfy following interfaces:\n```go\n// Counter describes a metric that accumulates values monotonically.\ntype Counter interface {\n\tWith(labelValues ...string) Counter\n\tAdd(delta float64)\n}\n\n// Gauge describes a metric that takes specific values over time.\ntype Gauge interface {\n\tWith(labelValues ...string) Gauge\n\tSet(value float64)\n\tAdd(delta float64)\n}\n\n// Histogram describes a metric that takes repeated observations of the same\n// kind of thing, and produces a statistical summary of those observations,\n// typically expressed as quantiles or buckets.\ntype Histogram interface {\n\tWith(labelValues ...string) Histogram\n\tObserve(value float64)\n}\n```\n\nYou can use one of `go-kit` implementations or try your own.\nFor instance, we want to expose metrics for _prometheus_ server.
