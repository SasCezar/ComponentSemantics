API\xe5\x9c\xa8`quickjs.h`\xe6\xa0\x87\xe5\xa4\xb4\xe4\xb8\xad\xe5\xae\x9a\xe4\xb9\x89\xe3\x80\x82\n\n#### 3.4.1 \xe8\xbf\x90\xe8\xa1\x8c\xe6\x97\xb6\xe5\x92\x8c\xe4\xb8\x8a\xe4\xb8\x8b\xe6\x96\x87\n\n`JSRuntime` represents a Javascript runtime corresponding to an object heap. Several runtimes can exist at the same time but they cannot exchange objects. Inside a given runtime, no multi-threading is supported.\n\n`JSContext` represents a Javascript context (or Realm). Each JSContext has its own global objects and system objects. There can be several JSContexts per JSRuntime and they can share objects, similary to frames of the same origin sharing Javascript objects in a web browser.\n\n#### 3.4.2 JSValue\n\n`JSValue` represents a Javascript value which can be a primitive type or an object. Reference counting is used, so it is important to explicitely duplicate (`JS_DupValue()`, increment the reference count) or free (`JS_FreeValue()`, decrement the reference count) JSValues.\n\n#### 3.4.3 C\xe5\x87\xbd\xe6\x95\xb0\n\nC functions can be created with `JS_NewCFunction()`. `JS_SetPropertyFunctionList()` is a shortcut to easily add functions, setters and getters properties to a given object.\n\nUnlike other embedded Javascript engines, there is no implicit stack, so C functions get their parameters as normal C parameters. As a general rule, C functions take constant `JSValue`s as parameters (so they don\xe2\x80\x99t need to free them) and return a newly allocated (=live) `JSValue`.\n\n#### 3.4.4 \xe9\x94\x99\xe8\xaf\xaf\xe5\xbc\x82\xe5\xb8\xb8\n\nExceptions: most C functions can return a Javascript exception. It must be explicitely tested and handled by the C code. The specific `JSValue` `JS_EXCEPTION` indicates that an exception occured. The actual exception object is stored in the `JSContext` and can be retrieved with `JS_GetException()`.\n\n#### 3.4.5 Script\xe4\xbb\xa3\xe7\xa0\x81\xe6\x89\xa7\xe8\xa1\x8c\n\nUse `JS_Eval()` to evaluate a script or module source.\n\nIf the script or module was compiled to bytecode with `qjsc`, `JS_EvalBinary()` achieves the same result. The advantage is that no compilation is needed so it is faster and smaller because the compiler can be removed from the executable if no `eval` is required.\n\nNote: the bytecode format is linked to a given QuickJS version. Moreover, no security check is done before its execution. Hence the bytecode should not be loaded from untrusted sources. That\xe2\x80\x99s why there is no option to output the bytecode to a binary file in `qjsc`.\n\n#### 3.4.6 JS\xe7\xb1\xbb\n\nC opaque data can be attached to a Javascript object. The type of the C opaque data is determined with the class ID (`JSClassID`) of the object. Hence the first step is to register a new class ID and JS class (`JS_NewClassID()`, `JS_NewClass()`). Then you can create objects of this class with `JS_NewObjectClass()` and get or set the C opaque point with `JS_GetOpaque()`/`JS_SetOpaque()`.\n\nWhen defining a new JS class, it is possible to declare a finalizer which is called when the object is destroyed. A `gc_mark` method can be provided so that the cycle removal algorithm can find the other objects referenced by this object. Other methods are available to define exotic object behaviors.\n\nThe Class ID are globally allocated (i.e. for all runtimes). The JSClass are allocated per `JSRuntime`. `JS_SetClassProto()` is used to define a prototype for a given class in a given JSContext. `JS_NewObjectClass()` sets this prototype in the created object.\n\nExamples are available in js\\_libc.c.\n\n#### 3.4.7 C\xe6\xa8\xa1\xe5\x9d\x97\n\n\xe6\x94\xaf\xe6\x8c\x81\xe5\x8a\xa8\xe6\x80\x81\xe6\x88\x96\xe8\x80\x85\xe9\x9d\x99\xe6\x80\x81\xe9\x93\xbe\xe6\x8e\xa5\xe7\x9a\x84\xe5\x8e\x9f\xe7\x94\x9fES6\xe6\xa8\xa1\xe5\x9d\x97\xe3\x80\x82\xe6\x9f\xa5\xe7\x9c\x8btest\\_bjson\xe5\x92\x8cbjson.so\xe7\xa4\xba\xe4\xbe\x8b\xe3\x80\x82\xe6\xa0\x87\xe5\x87\x86\xe5\xba\x93js\\_libc.c\xe4\xb9\x9f\xe6\x98\xaf\xe5\x8e\x9f\xe7\x94\x9f\xe6\xa8\xa1\xe5\x9d\x97\xe5\xbe\x88\xe5\xa5\xbd\xe7\x9a\x84\xe4\xb8\x80\xe4\xb8\xaa\xe4\xbe\x8b\xe5\xad\x90\xe3\x80\x82\n\n#### 3.4.8 \xe5\x86\x85\xe5\xad\x98\xe5\xa4\x84\xe7\x90\x86\n\n\xe4\xbd\xbf\xe7\x94\xa8 `JS_SetMemoryLimit()` \xe4\xb8\xba\xe7\xbb\x99\xe5\xae\x9a\xe7\x9a\x84JSRuntime\xe8\xae\xbe\xe7\xbd\xae\xe5\x85\xa8\xe5\xb1\x80\xe5\x86\x85\xe5\xad\x98\xe5\x88\x86\xe9\x85\x8d\xe9\x99\x90\xe5\x88\xb6\xe3\x80\x82\n\n`JS_NewRuntime2()`\xe5\x8f\xaf\xe4\xbb\xa5\xe6\x8f\x90\xe4\xbe\x9b\xe8\x87\xaa\xe5\xae\x9a\xe4\xb9\x89\xe5\x86\x85\xe5\xad\x98\xe5\x88\x86\xe9\x85\x8d\xe5\x8a\x9f\xe8\x83\xbd\xe3\x80\x82\n\n`JS_SetMaxStackSize()`\xe5\x8f\xaf\xe4\xbb\xa5\xe4\xbd\xbf\xe7\x94\xa8\xe8\xae\xbe\xe7\xbd\xae\xe6\x9c\x80\xe5\xa4\xa7\xe7\xb3\xbb\xe7\xbb\x9f\xe5\xa0\x86\xe6\xa0\x88\xe5\xa4\xa7\xe5\xb0\x8f\n\n#### 3.4.9 \xe6\x89\xa7\xe8\xa1\x8c\xe8\xb6\x85\xe6\x97\xb6\xe5\x92\x8c\xe4\xb8\xad\xe6\x96\xad\n\nUse `JS_SetInterruptHandler()` to set a callback which is regularly called by the engine when it is executing code. This callback can be used to implement an execution timeout.\n\n\xe5\x91\xbd\xe4\xbb\xa4\xe8\xa1\x8c\xe8\xa7\xa3\xe9\x87\x8a\xe5\x99\xa8\xe4\xbd\xbf\xe7\x94\xa8\xe5\xae\x83\xe6\x9d\xa5\xe5\xae\x9e\xe7\x8e\xb0 `Ctrl-C` \xe5\xa4\x84\xe7\x90\x86\xe7\xa8\x8b\xe5\xba\x8f\xe3\x80\x82\n\n4 \xe5\x86\x85\xe9\x83\xa8\xe5\xae\x9e\xe7\x8e\xb0\n-----------\n\n### 4.1 Bytecode\n\nThe compiler generates bytecode directly with no intermediate representation such as a parse tree, hence it is very fast. Several optimizations passes are done over the generated bytecode.\n\nA stack-based bytecode was chosen because it is simple and generates compact code.\n\nFor each function, the maximum stack size is computed at compile time so that no runtime stack overflow tests are needed.\n\nA separate compressed line number table is maintained for the debug information.\n\nAccess to closure variables is optimized and is almost as fast as local variables.\n\nDirect `eval` in strict mode is optimized.\n\n### 4.2 Executable generation\n\n#### 4.2.1 `qjsc` \xe7\xbc\x96\xe8\xaf\x91\xe5\x99\xa8\n\nThe `qjsc` compiler generates C sources from Javascript files. By default the C sources are compiled with the system compiler (`gcc` or `clang`).\n\nThe generated C source contains the bytecode of the compiled functions or modules. If a full complete executable is needed, it also contains a `main()` function with the necessary C code to initialize the Javascript engine and to load and execute the compiled functions and modules.\n\nJavascript code can be mixed with C modules.\n\nIn order to have smaller executables, specific Javascript features can be disabled, in particular `eval` or the regular expressions. The code removal relies on the Link Time Optimization of the system compiler.\n\n#### 4.2.2 \xe4\xba\x8c\xe8\xbf\x9b\xe5\x88\xb6 JSON\n\n`qjsc` works by compiling scripts or modules and then serializing them to a binary format. A subset of this format (without functions or modules) can be used as binary JSON. The example test\\_bjson.js shows how to use it.\n\nWarning: the binary JSON format may change without notice, so it should not be used to store persistent data. The test\\_bjson.js example is only used to test the binary object format functions.\n\n### 4.3 \xe8\xbf\x90\xe8\xa1\x8c\xe6\x97\xb6\n\n#### 4.3.1 Strings\n\n\xe5\xad\x97\xe7\xac\xa6\xe4\xb8\xb2\xe5\xad\x98\xe5\x82\xa8\xe4\xb8\xba8\xe4\xbd\x8d\xe6\x88\x9616\xe4\xbd\x8d\xe5\xad\x97\xe7\xac\xa6\xe6\x95\xb0\xe7\xbb\x84\xe3\x80\x82\xe5\x9b\xa0\xe6\xad\xa4\xef\xbc\x8c\xe9\x9a\x8f\xe6\x9c\xba\xe8\xae\xbf\xe9\x97\xae\xe5\xad\x97\xe7\xac\xa6\xe6\x80\xbb\xe6\x98\xaf\xe5\xbe\x88\xe5\xbf\xab\xe3\x80\x82\n\nC API\xe6\x8f\x90\xe4\xbe\x9b\xe5\xb0\x86Javascript\xe5\xad\x97\xe7\xac\xa6\xe4\xb8\xb2\xe8\xbd\xac\xe6\x8d\xa2\xe4\xb8\xbaC UTF-8\xe7\xbc\x96\xe7\xa0\x81\xe5\xad\x97\xe7\xac\xa6\xe4\xb8\xb2\xe7\x9a\x84\xe5\x87\xbd\xe6\x95\xb0\xe3\x80\x82\xe6\x9c\x80\xe5\xb8\xb8\xe8\xa7\x81\xe6\x83\x85\xe5\x86\xb5\xe6\x98\xaf Javascript\xe5\xad\x97\xe7\xac\xa6\xe4\xb8\xb2\xe4\xbb\x85\xe5\x8c\x85\xe5\x90\xabASCII \xe5\xad\x97\xe7\xac\xa6\xe4\xb8\xb2\xe4\xb8\x8d\xe6\xb6\x89\xe5\x8f\x8a\xe5\xa4\x8d\xe5\x88\xb6\xe3\x80\x82\n\n#### 4.3.2 Objects\n\nThe object shapes (object prototype, property names and flags) are shared between objects to save memory.\n\nArrays with no holes (except at the end of the array) are optimized.\n\nTypedArray\xe8\xae\xbf\xe9\x97\xae\xe5\xb7\xb2\xe4\xbc\x98\xe5\x8c\x96\xe3\x80\x82\n\n#### 4.3.3 Atoms\n\nObject property names and some strings are stored as Atoms (unique strings) to save memory and allow fast comparison. Atoms are represented as a 32 bit integer. Half of the atom range is reserved for immediate integer literals from _0_ to _2^{31}-1_.\n\n#### 4.3.4 Numbers\n\nNumbers are represented either as 32-bit signed integers or 64-bit IEEE-754 floating point values. Most operations have fast paths for the 32-bit integer case.\n\n#### 4.3.5 \xe5\x9e\x83\xe5\x9c\xbe\xe5\x9b\x9e\xe6\x94\xb6\n\n\xe5\xbc\x95\xe7\x94\xa8\xe8\xae\xa1\xe6\x95\xb0\xe7\x94\xa8\xe4\xba\x8e\xe8\x87\xaa\xe5\x8a\xa8\xe5\x92\x8c\xe5\x87\x86\xe7\xa1\xae\xe5\x9c\xb0\xe9\x87\x8a\xe6\x94\xbe\xe5\xaf\xb9\xe8\xb1\xa1\xe3\x80\x82A separate cycle removal pass is done when the allocated memory becomes too large. The cycle removal algorithm only uses the reference counts and the object content, so no explicit garbage collection roots need to be manipulated in the C code.\n\n#### 4.3.6 JSValue\n\nIt is a Javascript value which can be a primitive type (such as Number, String, ...) or an Object. NaN boxing is used in the 32-bit version to store 64-bit floating point numbers. The representation is optimized so that 32-bit integers and reference counted values can be efficiently tested.\n\nIn 64-bit code, JSValue are 128-bit large and no NaN boxing is used. The rationale is that in 64-bit code memory usage is less critical.\n\nIn both cases (32 or 64 bits), JSValue exactly fits two CPU registers, so it can be efficiently returned by C functions.\n\n####
